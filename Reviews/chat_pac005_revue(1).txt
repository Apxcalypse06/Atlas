## Blocking issues

### 1. Import non transactionnel : une erreur mémoire peut laisser une capsule partiellement modifiée

**Problème**

`PacSessionImporter.import_session()` écrit d’abord le fichier de session, `sessions/index.json`, le changelog et le manifeste, puis appelle seulement ensuite `PacMemoryManager().add(...)` :

- écriture session/index/changelog/manifest : `src/pac/session.py:63`, `src/pac/session.py:92-132`
- ajout mémoire ensuite : `src/pac/session.py:65-72`

Si l’ajout mémoire échoue, l’import retourne une erreur, mais la capsule a déjà été modifiée. Cela viole l’exigence d’intégrité et de cohérence des fichiers modifiés.

**Reproduction vérifiée**

J’ai créé une capsule, corrompu uniquement `user/memory.json`, puis exécuté :

```bash
pac validate partial.pac.zip
pac session import chat.txt --capsule partial.pac.zip
```

Résultat observé :

```text
pac validate => Result: VALID
pac session import => Error: user/memory.json is invalid JSON ...
```

Mais après l’échec, le ZIP contenait déjà :

```text
sessions/SES-...txt
sessions/index.json
logs/changelog.md avec entrée d’import
user/memory.json toujours invalide
```

Donc la commande retourne `1`, mais laisse une trace d’import partielle sans entrée mémoire correspondante.

**Correction obligatoire avant DONE**

L’import doit devenir atomique au niveau PAC-005. Options acceptables :

1. prévalider tous les fichiers nécessaires, y compris `user/memory.json`, avant toute écriture ;
2. préparer session + index + changelog + manifeste + mémoire en mémoire, puis effectuer une seule réécriture ZIP atomique ;
3. ou mettre en place un rollback fiable si la seconde phase échoue.

Ajouter un test obligatoire : échec de l’ajout mémoire ⇒ aucune modification persistée dans la capsule.

---

### 2. `--force` peut produire des tracebacks utilisateur sur capsules structurellement invalides

**Problème**

Le CLI ne capture que `ValueError` et `FileNotFoundError` pour `pac session import` :

```python
except (ValueError, FileNotFoundError) as exc:
```

Voir `src/pac/cli.py:103-105`.

Avec `--force`, une capsule invalide peut dépasser la validation initiale, puis échouer plus bas avec des exceptions non converties en message utilisateur propre :

- ZIP sans `MANIFEST.json` ⇒ `KeyError: 'MANIFEST.json'` à `src/pac/session.py:98`
- fichier non-ZIP avec `--force` ⇒ `zipfile.BadZipFile` à `src/pac/session.py:92`

**Reproduction vérifiée**

```bash
pac session import chat.txt --capsule bad.pac.zip --force
```

sur un ZIP lisible mais sans `MANIFEST.json` produit un traceback Python au lieu d’un message `Error: ...`.

**Correction obligatoire avant DONE**

`--force` ne doit pas signifier “continuer sur n’importe quelle archive non exploitable”. Il faut :

- distinguer capsule invalide mais structurellement exploitable vs archive non-PAC/non-ZIP/corrompue ;
- convertir `KeyError`, `zipfile.BadZipFile`, erreurs JSON de manifeste, etc. en `ValueError` métier ou en rapport d’erreur contrôlé ;
- garantir un code retour `1` avec message utilisateur propre, sans traceback.

Ajouter des tests CLI pour `--force` sur :

- non-ZIP ;
- ZIP sans `MANIFEST.json` ;
- capsule invalide mais structurellement PAC.

## Non-blocking improvements

### Tests

La suite passe, mais la couverture PAC-005 reste minimale. À ajouter après les blockers :

- assertion que le SHA-256 stocké est exactement égal au hash du fichier importé ;
- assertion explicite du changelog ;
- test de multiples imports et append dans `sessions/index.json`;
- test du fallback `label = source.name` ;
- test d’import d’un fichier sans suffixe ;
- test `zipfile.testzip()` après import ;
- test `--force` sur capsule invalide mais réparable/exploitable.

### Documentation

La documentation QA est plus complète que la documentation embarquée dans l’implémentation.

À aligner :

- `docs/SESSION_IMPORT_COMMAND.md` dans l’implémentation ne documente pas `--force` ni les codes de retour, alors que la QA archive les mentionne.
- `README.md` montre `--label`, mais pas `--force`.
- `sessions/index.json` utilise `"version": "0.1"` alors que le package est PAC v0.2 / PAC-005. Ce n’est pas bloquant, mais doit être documenté ou aligné.

### Métadonnée mémoire

L’entrée mémoire créée par `PacMemoryManager` a actuellement :

```json
"source": "pac memory add"
```

Même lorsqu’elle est créée par `pac session import`. Ce n’est pas bloquant, car la mémoire est bien ajoutée, mais une future amélioration serait de permettre à `PacMemoryManager` de recevoir une source explicite, par exemple `"pac session import"`.

### Packaging release professionnelle

Le packaging est suffisant pour une alpha editable, mais une release professionnelle devrait ajouter au minimum :

- licence ;
- auteurs/maintainers ;
- `readme` dans `pyproject.toml`;
- classifiers ;
- dépendances de dev ou extras de test ;
- stratégie de versionnement/release notes.
